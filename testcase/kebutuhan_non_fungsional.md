# Dokumen Kebutuhan Non-Fungsional: Usability & Reliability
## Proyek: CivadTA (Sistem E-Commerce & Pengembalian Buku)

Dokumen ini mendefinisikan **Kebutuhan Non-Fungsional (Non-Functional Requirements / NFR)** untuk aplikasi CivadTA. Fokus utama dari dokumen ini adalah pada dua kategori kualitas sistem, yaitu **Usability (Kemudahan Penggunaan)** dan **Reliability (Keandalan)**, yang disesuaikan dengan karakteristik teknis dan fungsional CivadTA (seperti checkout buku, integrasi Midtrans, pengunggahan video bukti pengembalian 50MB, ulasan, dan chat).

---

## 1. Kategori: Usability (Kemudahan Penggunaan)
Kebutuhan Usability memastikan bahwa aplikasi CivadTA mudah dipahami, dipelajari, digunakan, dan memberikan pengalaman pengguna yang menyenangkan bagi pelanggan maupun administrator.

| ID NFR | Deskripsi Kebutuhan | Parameter Keberhasilan / Metrik | Justifikasi Skenario Proyek |
| :--- | :--- | :--- | :--- |
| **NFR-USA-001** | **Desain Antarmuka Responsif (Responsive Web Design)** | Aplikasi harus dapat diakses secara optimal dengan tata letak yang menyesuaikan otomatis pada perangkat Mobile (lebar min. 320px), Tablet (lebar min. 768px), dan Desktop (lebar min. 1024px) tanpa ada elemen teks atau tombol yang saling tumpang tindih. | Pelanggan umumnya mencari katalog dan melakukan pemesanan buku melalui smartphone, sedangkan Admin mengelola pesanan melalui desktop. |
| **NFR-USA-002** | **Kemudahan Navigasi & Efisiensi Alur Pembelian** | Pengguna baru harus dapat menemukan buku, memasukkannya ke keranjang, dan menyelesaikan proses checkout dalam waktu kurang dari 3 menit atau maksimum 5 kali klik dari halaman utama (*landing page*). | Meminimalisir tingkat pembatalan transaksi (*cart abandonment rate*) karena proses checkout yang rumit. |
| **NFR-USA-003** | **Pencegahan & Umpan Balik Kesalahan Validasi secara Real-Time** | Pesan kesalahan (*error validation*) harus muncul secara langsung di bawah kolom input terkait (seperti pendaftaran akun dan pengisian nomor rekening) sesaat setelah pengguna memindahkan kursor (*on blur* / real-time) dengan bahasa Indonesia yang jelas dan instruktif. | Menghindari pengguna mengirimkan formulir berulang kali untuk kesalahan kecil (contoh: format nomor rekening tidak valid atau password kurang dari 8 karakter). |
| **NFR-USA-004** | **Kejelasan Petunjuk Batasan Upload Dokumen (Return)** | Form pengajuan pengembalian buku harus secara eksplisit mencantumkan petunjuk batas ukuran file (maks. 50 MB) dan format video yang didukung (`.mp4`, `.mov`, `.avi`, `.webm`). Sistem harus memberikan indikator proses (*loading spinner* atau *progress bar*) saat video sedang diunggah. | Mengunggah video berukuran besar membutuhkan waktu. Indikator visual mencegah pengguna mengklik tombol kirim berkali-kali atau menutup halaman sebelum proses selesai. |
| **NFR-USA-005** | **Aksesibilitas Kontras Warna & Tipografi** | Kontras warna teks dengan latar belakang (*contrast ratio*) harus memenuhi standar minimal WCAG 2.1 AA (minimal 4.5:1 untuk teks biasa) dan menggunakan font sans-serif yang mudah dibaca dengan ukuran minimal 12px pada perangkat mobile. | Memastikan semua pengguna, termasuk yang memiliki gangguan penglihatan ringan atau menggunakan perangkat di bawah sinar matahari langsung, dapat membaca informasi buku dengan jelas. |
| **NFR-USA-006** | **Konsistensi Status Pesanan & Notifikasi** | Setiap perubahan status pesanan (misal: dari "Pembayaran Berhasil" ke "Pesanan Sedang Dikemas") dan pengajuan retur (dari "Pending" ke "Disetujui/Ditolak") harus ditampilkan dengan label warna yang konsisten (cth: Hijau untuk sukses, Kuning untuk pending, Merah untuk ditolak) di halaman riwayat pelanggan. | Konsistensi visual memudahkan pelanggan melacak pesanan mereka tanpa kebingungan. |

## 2. Kategori: Reliability (Keandalan)
Kebutuhan Reliability memastikan sistem CivadTA tetap stabil, dapat diandalkan, aman dari kegagalan transaksi, dan mampu pulih dari kondisi eror/kegagalan sistem secara cepat.

| ID NFR | Deskripsi Kebutuhan | Parameter Keberhasilan / Metrik | Justifikasi Skenario Proyek |
| :--- | :--- | :--- | :--- |
| **NFR-REL-001** | **Ketersediaan Layanan (Uptime/Availability)** | Aplikasi CivadTA harus memiliki tingkat ketersediaan (*uptime*) minimal **99.5%** dalam satu bulan (maksimal waktu mati/*downtime* tidak terencana adalah ~3.6 jam per bulan). | Sistem e-commerce harus selalu siap menerima pesanan dari pelanggan kapan saja (24/7). |
| **NFR-REL-002** | **Keandalan Transaksi Pembayaran (Transaction Integrity)** | Pembayaran menggunakan Midtrans Payment Gateway harus menerapkan mekanisme *webhook* callback yang andal dengan status sinkronisasi otomatis. Jika terjadi kegagalan jaringan saat callback, sistem harus mampu melakukan proses rekonsiliasi ulang otomatis saat koneksi pulih tanpa terjadinya duplikasi status bayar. | Kegagalan sinkronisasi status pembayaran dapat menyebabkan kerugian finansial atau ketidakpuasan pelanggan (buku tidak dikirim padahal saldo sudah terpotong). |
| **NFR-REL-003** | **Ketahanan Pengunggahan File Besar (Upload Robustness)** | Sistem harus mampu menangani pengunggahan video bukti pengembalian hingga 50 MB secara stabil. Backend Laravel harus dikonfigurasi (`upload_max_filesize` dan `post_max_size` di php.ini/nginx) agar tidak mengalami *Request Timeout* (HTTP 504) atau *Internal Server Error* (HTTP 500) pada koneksi dengan kecepatan unggah standar (min. 5 Mbps). | Fitur retur buku mewajibkan bukti video. Jika server gagal memproses file besar ini, pengguna tidak akan bisa melakukan pengembalian barang. |
| **NFR-REL-004** | **Konsistensi Data Transaksi (Database Atomicity)** | Setiap proses checkout pesanan dan pengurangan stok buku harus menggunakan mekanisme database *transaction* (`DB::beginTransaction`). Jika terjadi kegagalan pada salah satu langkah checkout (misal: gagal menyimpan data barang pesanan), seluruh operasi database terkait transaksi tersebut harus dibatalkan (*rollback*). | Menghindari kondisi inkonsistensi data seperti stok buku berkurang namun data transaksi pemesanan tidak tercatat di database. |
| **NFR-REL-005** | **Pencegahan Pengajuan Retur Ganda (Idempotency)** | Sistem harus mengunci status pesanan saat pengajuan pengembalian pertama dikirim. Permintaan POST berulang ke `/pelanggan/pengembalian/simpan` untuk `order_id` yang sama harus ditolak secara absolut di tingkat backend menggunakan validasi database unik atau kunci transaksi. | Mencegah celah manipulasi di mana satu transaksi pemesanan dapat mengklaim pengembalian dana berkali-kali secara simultan. |
| **NFR-REL-006** | **Keandalan Pengiriman Pesan Chat** | Fitur chat antara pelanggan dan admin harus menjamin pengiriman pesan dengan tingkat keberhasilan **99%**. Jika koneksi internet terputus saat mengirim pesan, sistem harus menampilkan status "Gagal Terkirim" dan menyediakan tombol "Kirim Ulang" (*Retry*), bukan menghilangkannya secara diam-diam. | Chat adalah sarana komunikasi krusial untuk negosiasi retur dan pertanyaan stok buku. |
| **NFR-REL-007** | **Pencadangan Data Otomatis & Pemulihan (Backup & Disaster Recovery)** | Database sistem harus dicadangkan secara otomatis setiap hari pada jam non-sibuk (cth: pukul 02.00 WIB). Waktu pemulihan data (*Recovery Time Objective / RTO*) pasca kerusakan database tidak boleh melebihi 2 jam, dengan kehilangan data maksimal (*Recovery Point Objective / RPO*) adalah 24 jam. | Melindungi data penting seperti informasi pelanggan, histori transaksi keuangan, dan ulasan produk dari kehilangan permanen jika server mengalami kerusakan fisik. |
